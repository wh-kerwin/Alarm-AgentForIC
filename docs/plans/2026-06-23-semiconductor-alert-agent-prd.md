# 半导体异常告警与根因分析 Agent PRD

## 1. 背景与问题

半导体产线中的设备、工艺、良率系统会持续产生大量告警。当前工程师在处理告警时，通常需要手动登录多个系统，收集 alarm log、设备状态、recipe、sensor trace、SPC/FDC 指标、lot/wafer 历史、维护记录、缺陷与良率数据，再凭经验判断可能根因并决定处置动作。

这个过程存在几个典型问题：

- 数据分散在 EAP、MES、FDC、SPC、APC、YMS、设备日志、维修系统和知识库中。
- 告警之间存在时间、设备、recipe、lot、chamber、产品族等关联，但人工关联成本高。
- 初级工程师依赖资深工程师经验，响应时间和处置质量不稳定。
- 重复告警、误报、连锁告警会放大排查成本。
- 根因分析结果没有结构化沉淀，后续相似问题仍然重复排查。

本产品目标是构建一个半导体行业垂直告警 Agent：设备或系统产生异常告警后，Agent 自动收集上下文数据、关联相关事件、生成可能根因排序、给出处置建议，并支持工程师确认、反馈和沉淀知识，最终减少响应时间和提高排查一致性。

## 2. 产品定位

本产品不是单纯的聊天机器人，也不是替代 FDC/SPC/MES 的监控系统，而是一个面向半导体制造场景的异常告警分析与处置协同 Agent。

产品定位：

- 第一阶段：设备告警响应 Agent。
- 第二阶段：良率与缺陷异常分析 Agent。
- 第三阶段：工艺参数异常分析 Agent。
- 最终形态：通用异常告警 Agent 平台，支持多类告警、多源数据、多场景根因分析和闭环处置。

核心价值：

- 把告警从“通知”升级为“可解释、可处置、可追踪的分析任务”。
- 把工程师经验转化为可复用的规则、案例和知识。
- 把多系统数据查询转化为 Agent 自动编排。
- 把根因分析从一次性口头判断转化为可审计的结构化结论。

## 3. 目标用户

### 3.1 一线设备工程师

关注设备 alarm、chamber 状态、sensor trace、PM 记录、备件更换、停机影响。需要快速判断是否需要停机、重跑、切机、通知资深工程师或执行标准处置。

### 3.2 工艺工程师

关注 recipe、工艺参数、SPC/FDC 异常、lot/wafer 影响范围、异常是否与工艺窗口偏移有关。

### 3.3 良率工程师

关注 yield drop、defect pattern、wafer map、metrology trend、异常批次和历史相似案例。

### 3.4 值班主管/Shift Lead

关注异常优先级、影响范围、处置进展、升级路径、是否影响 WIP、产能和交付。

### 3.5 制造/IT/自动化团队

关注系统集成、数据权限、审计、安全、接口稳定性和 Agent 运行可控性。

## 4. 产品目标

### 4.1 业务目标

- 缩短告警响应时间。
- 降低工程师跨系统查询成本。
- 提高初步根因判断的一致性。
- 降低重复告警和误报带来的排查负担。
- 沉淀可复用的异常案例库和处置知识库。
- 支持后续从设备告警扩展到良率、工艺和综合制造异常。

### 4.2 MVP 目标

MVP 聚焦设备维护/报警响应场景，选择 1 到 2 类高频设备或高价值设备族作为试点。Agent 在收到设备 alarm 后，能够自动完成：

- 识别告警类型、设备、chamber、时间、lot、recipe。
- 拉取告警前后指定时间窗口内的相关数据。
- 关联近期维护、recipe 切换、sensor 异常、SPC/FDC 规则触发、同设备历史告警。
- 输出可能根因 Top 3。
- 输出建议处置动作、风险提示和升级建议。
- 允许工程师确认、驳回、补充原因和记录最终处置。

## 5. 成功指标

### 5.1 效率指标

- 平均首次响应时间降低 30% 以上。
- 工程师手动查询系统次数降低 50% 以上。
- 常见告警初步分析时间从分钟级/十几分钟级降低到 1 到 3 分钟。

### 5.2 质量指标

- Top 3 根因命中率达到 60% 以上，后续通过反馈持续提升。
- 标准处置建议被采纳率达到 50% 以上。
- 重复告警识别准确率达到 70% 以上。

### 5.3 运营指标

- 每条告警分析均有可追溯的数据证据链。
- 每次人工反馈都能沉淀到案例库或规则优化流程。
- 高优先级告警升级路径明确，避免无人处理。

1. 作为一线设备工程师，我希望设备报警后自动看到相关上下文数据 so that 我不用手动登录多个系统排查。

5. 作为一线设备工程师，我希望 Agent 给出根因排序而不是单一结论 so that 我能结合现场经验进行判断。
6. 作为一线设备工程师，我希望每个根因都有证据说明 so that 我能信任或反驳 Agent 的判断。
7. 作为一线设备工程师，我希望 Agent 推荐标准处置动作 so that 我能快速执行 SOP。
8. 作为一线设备工程师，我希望能一键记录最终根因和处置动作 so that 后续相似告警可以复用。
9. 作为工艺工程师，我希望看到告警相关 lot、recipe 和关键参数趋势 so that 我能判断是否存在工艺窗口偏移。
10. 作为工艺工程师，我希望 Agent 关联 SPC/FDC 规则触发情况 so that 我能判断 alarm 是否只是结果，而不是根因。
11. 作为良率工程师，我希望 Agent 在良率异常时关联 wafer map、defect pattern 和 metrology 数据 so that 我能快速缩小影响范围。
12. 作为良率工程师，我希望 Agent 找出异常批次与正常批次的关键差异 so that 我能识别潜在根因。
13. 作为值班主管，我希望看到告警影响范围和优先级 so that 我能决定是否升级和调配资源。
14. 作为值班主管，我希望看到告警处理状态、负责人和预计恢复时间 so that 我能管理班次风险。
15. 作为制造管理者，我希望查看告警统计和复发趋势 so that 我能推动系统性改善。
16. 作为知识库管理员，我希望把确认过的案例沉淀为标准案例 so that Agent 后续推荐更准确。
17. 作为 IT/自动化工程师，我希望 Agent 的数据访问有权限控制和审计日志 so that 系统符合工厂安全要求。
18. 作为系统管理员，我希望可以配置不同告警类型的数据收集策略 so that 不同场景使用不同上下文。
19. 作为系统管理员，我希望可以禁用高风险自动动作 so that Agent 只做建议，不越权控制产线。
20. 作为业务负责人，我希望 Agent 的效果可量化 so that 我能判断是否继续扩展到更多设备和工艺场景。

## 7. 功能范围

### 7.1 告警接入

系统需要支持从 EAP、MES、FDC/SPC、设备日志系统或消息队列接入告警。每条告警至少需要包含：

- 告警 ID。
- 告警来源系统。
- 告警类型和严重等级。
- 设备 ID。
- chamber/module ID，如果适用。
- 发生时间。
- lot/wafer/product/recipe 信息，如果可用。
- 原始告警文本。

### 7.2 告警归一化

不同系统的告警格式需要统一成标准 Alert Event。归一化后，Agent 才能复用同一套分析流程。

标准 Alert Event 包含：

- alert_id
- source
- severity
- equipment_id
- chamber_id
- timestamp
- alarm_code
- alarm_message
- lot_id
- wafer_id
- recipe_id
- product_id
- current_state
- raw_payload

### 7.3 上下文自动收集

Agent 根据告警类型和设备类型，自动生成数据收集计划。MVP 阶段建议先支持以下数据：

- 告警前后 30 到 120 分钟 alarm log。
- 设备状态变化记录。
- sensor trace 或关键 FDC 参数。
- recipe 切换记录。
- lot/wafer 在当前设备的加工历史。
- 最近 PM、维修、部件更换记录。
- 同设备/同 chamber 历史相似告警。
- 相关 SOP、OCAP、工程备注。

### 7.4 事件关联

Agent 需要把收集到的数据转化为结构化事件，并按时间线和关系图组织。

事件类型包括：

- AlarmEvent：告警事件。
- StateChangeEvent：设备状态变化。
- ProcessRunEvent：加工事件。
- RecipeChangeEvent：recipe 切换。
- SensorAnomalyEvent：传感器异常。
- MaintenanceEvent：维护事件。
- SPCFDCViolationEvent：规则触发事件。
- YieldDefectEvent：良率或缺陷事件。
- HumanActionEvent：人工处置记录。

### 7.5 根因候选生成

Agent 需要基于规则、统计相似度、知识库检索和大模型推理生成根因候选。每个候选根因必须包含：

- 根因描述。
- 置信度或优先级。
- 支持证据。
- 反证或不确定性。
- 影响范围。
- 建议验证动作。
- 建议处置动作。

MVP 不要求 Agent 给出唯一正确答案，而是要求给出清晰的 Top 3 候选根因和证据链。

### 7.6 处置建议

Agent 根据根因候选、SOP、OCAP 和历史案例生成处置建议。建议分为：

- 立即检查项。
- 安全确认项。
- 标准恢复动作。
- 是否 hold lot。
- 是否切换设备/chamber。
- 是否通知设备/工艺/良率工程师。
- 是否升级到主管或资深工程师。

MVP 阶段 Agent 只提供建议，不直接执行停机、放行、改 recipe、改参数等高风险动作。

### 7.7 人工反馈与闭环

工程师可以对 Agent 输出进行反馈：

- 采纳某个根因。
- 标记根因错误。
- 补充真实根因。
- 补充处置动作。
- 标记是否复发。
- 将案例沉淀为知识库条目。

这些反馈用于优化案例检索、规则配置和后续模型评估。

### 7.8 告警工作台

前端需要提供一个告警处理工作台，而不是普通聊天窗口。核心页面包括：

- 告警列表：按严重等级、设备、状态、负责人、时间筛选。
- 告警详情：展示原始告警、影响范围、状态和负责人。
- 时间线视图：展示告警前后的事件序列。
- 根因分析视图：展示 Top 3 根因、证据、反证和建议验证动作。
- 处置建议视图：展示推荐 SOP、检查项和升级建议。
- 反馈表单：记录最终原因和处置结果。
- 案例库：检索历史相似告警和处置记录。

## 8. 分阶段路线

### 8.1 阶段一：设备告警响应 Agent

目标是解决设备 alarm 后工程师响应慢、跨系统查数据多、经验不可复用的问题。

优先能力：

- 接入设备 alarm。
- 自动拉取 alarm log、设备状态、sensor/FDC 数据、recipe、PM 记录和历史相似告警。
- 生成时间线。
- 输出 Top 3 设备侧根因。
- 推荐处置动作。
- 记录工程师反馈。

典型根因类别：

- 传感器读数异常或漂移。
- chamber 状态异常。
- recipe 参数不匹配。
- PM 后状态不稳定。
- 部件更换后校准问题。
- 前序 lot 或异常 wafer 造成连锁影响。
- 重复告警或误报。

### 8.2 阶段二：良率与缺陷异常分析 Agent

目标是从设备告警扩展到良率异常、defect excursion 和 wafer map 异常。

新增能力：

- 接入 YMS、defect inspection、metrology、wafer map 数据。
- 关联异常 lot 与设备、recipe、时间窗口、chamber、产品族。
- 对比正常批次与异常批次的差异。
- 检索历史相似 defect pattern。
- 输出可能影响路径，例如设备、工艺站点、材料、recipe 或检测异常。

典型根因类别：

- 特定设备/chamber 导致异常。
- 特定 recipe 或产品族敏感。
- 前后道工艺交互导致缺陷。
- metrology 或 inspection 本身异常。
- 材料批次或来料异常。

### 8.3 阶段三：工艺参数异常分析 Agent

目标是支持 SPC/FDC/APC 异常、工艺漂移、参数越界和 process window 偏移分析。

新增能力：

- 接入 SPC rule、FDC trace、APC 调参记录。
- 支持参数趋势、漂移检测、异常窗口定位。
- 关联工艺参数与设备状态、recipe、lot 和良率结果。
- 识别异常参数之间的相关性。
- 输出验证实验或工程检查建议。

典型根因类别：

- 参数长期漂移。
- recipe 或 setpoint 修改影响。
- APC 调整引入偏差。
- sensor calibration 问题。
- 环境或耗材变化影响。
- 工艺窗口边缘运行。

### 8.4 最终形态：通用异常告警 Agent 平台

当设备、良率、工艺三类场景都跑通后，产品抽象成通用告警 Agent 平台。

平台化能力：

- 通用告警事件模型。
- 可配置数据收集策略。
- 可配置事件关联规则。
- 可配置根因分类体系。
- 可配置处置建议模板。
- 多 Agent 协作，例如设备 Agent、工艺 Agent、良率 Agent、知识库 Agent。
- 多场景统一工作台。
- 效果评估和持续学习闭环。

## 9. Agent 架构设计

### 9.1 核心模块

#### Alert Ingestion

负责接入多源告警，并转换成标准 Alert Event。

#### Context Collector

根据告警类型、设备类型和场景配置，自动查询相关系统并收集上下文数据。

#### Event Correlator

把上下文数据标准化为事件，并生成时间线、关系图和候选关联。

#### Root Cause Analyzer

综合规则、案例检索、统计分析和 LLM 推理，生成候选根因和证据链。

#### Recommendation Engine

结合 SOP、OCAP、历史案例和权限策略，生成处置建议。

#### Knowledge Base

存储 SOP、OCAP、历史案例、设备知识、工艺知识、常见问题和人工反馈。

#### Human Feedback Loop

收集工程师对根因和处置建议的确认、驳回和补充，支持后续评估与优化。

#### Alert Workbench

提供告警列表、详情、时间线、根因、建议、反馈、案例检索等前端操作界面。

### 9.2 推荐技术路线

第一版建议采用“规则/检索/LLM 混合架构”，不要只依赖大模型。

- 规则：用于告警分级、数据收集策略、强约束判断、SOP 匹配。
- 检索：用于查找历史案例、SOP、OCAP、设备手册、工程备注。
- 统计/算法：用于趋势、异常点、相似批次、参数漂移、相关性分析。
- LLM：用于跨源信息总结、根因候选组织、证据链解释和处置建议生成。

原因：

- 半导体场景对可解释性和可审计性要求高。
- 数据质量和接口稳定性不一定一开始就完美。
- 工程师需要看到证据，而不是只看到自然语言结论。
- 高风险动作必须由规则和权限系统约束。

## 10. 数据流

1. 告警产生：设备、FDC/SPC、MES 或其他系统产生异常告警。
2. 告警接入：Alert Ingestion 将告警转为标准 Alert Event。
3. 场景识别：系统识别该告警属于设备、良率、工艺或综合异常。
4. 数据计划：Context Collector 根据策略生成数据收集清单。
5. 数据查询：系统查询日志、MES、FDC、SPC、PM、YMS、知识库等数据。
6. 事件建模：Event Correlator 将数据转为标准事件。
7. 关联分析：系统构建时间线、相似案例、影响范围和关键异常点。
8. 根因生成：Root Cause Analyzer 生成 Top 3 根因候选。
9. 建议生成：Recommendation Engine 输出验证动作、处置建议和升级路径。
10. 人工确认：工程师采纳、驳回或修正根因与处置建议。
11. 知识沉淀：反馈写入案例库和评估数据集。
12. 持续优化：根据反馈优化规则、检索和模型提示词/评估集。

## 12. 权限与安全边界

MVP 阶段必须明确 Agent 权限：

- Agent 可以读取授权数据。
- Agent 可以生成分析和建议。
- Agent 可以创建处置记录草稿。
- Agent 不可以自动放行 lot。
- Agent 不可以自动修改 recipe。
- Agent 不可以自动修改设备参数。
- Agent 不可以自动关闭高等级告警。
- Agent 不可以绕过工程师确认执行高风险动作。

所有查询、分析、建议和人工确认都需要留审计日志。

## 13. 非功能需求

### 13.1 可解释性

每个根因候选必须有证据链。没有证据的推测只能标记为低置信度。

### 13.2 可追溯性

每次分析需要保留输入数据版本、查询时间、知识库命中内容、模型输出和人工反馈。

### 13.3 响应时间

MVP 目标是在告警触发后 1 到 3 分钟内给出首轮分析结果。复杂数据可以异步补全。

### 13.4 数据权限

不同角色只能访问授权范围内的设备、产品、lot 和工艺数据。

### 13.5 可配置性

不同设备族、工艺站点和告警类型需要可配置不同的数据收集策略和处置模板。

### 13.6 可靠性

某个数据源不可用时，Agent 需要降级输出，并明确说明缺失数据对分析结论的影响。

## 14. MVP 范围

### 14.1 包含

- 设备告警接入。
- 标准 Alert Event 模型。
- 设备告警工作台。
- 告警上下文自动收集。
- 时间线生成。
- 历史相似告警检索。
- Top 3 根因候选。
- 处置建议。
- 人工反馈闭环。
- 基础效果看板。

### 14.2 不包含

- 自动控制设备。
- 自动修改 recipe 或参数。
- 全厂所有设备一次性覆盖。
- 复杂因果模型训练。
- 完整数字孪生。
- 替代工程师做最终处置决策。
- 跨厂区统一部署。

## 15. 第一版试点建议

第一版不要从全场景开始，建议选择一个满足以下条件的设备族或告警类型：

- 告警频率高。
- 工程师排查耗时长。
- 数据源相对可获取。
- SOP 或历史案例较多。
- 根因类别相对稳定。
- 业务方愿意提供反馈。

试点路径：

1. 选择 1 个设备族和 5 到 10 个高频告警代码。
2. 整理这些告警的历史案例、SOP、OCAP 和关键数据源。
3. 定义每个告警的数据收集策略。
4. 建立根因分类和处置建议模板。
5. 上线只读分析工作台。
6. 收集 2 到 4 周工程师反馈。
7. 根据命中率和采纳率决定是否扩展。

## 16. 关键风险

### 16.1 数据源接入困难

半导体工厂系统多、接口复杂、权限严格。需要优先选择可接入数据源，不要一开始依赖所有系统。

### 16.2 数据质量不稳定

设备日志、人工维修记录、recipe 变更记录可能不完整。Agent 必须明确缺失数据，不得伪造结论。

### 16.3 工程师不信任

如果只给结论不给证据，工程师不会采纳。必须把证据链、反证和不确定性作为核心产品能力。

### 16.4 大模型幻觉

LLM 只能在检索和结构化数据基础上组织分析，不能凭空编造设备知识、SOP 或处置动作。

### 16.5 责任边界

Agent 在 MVP 阶段只做辅助决策，最终操作由授权工程师确认。

## 17. 验收标准

MVP 可以按以下标准验收：

- 支持至少 1 个设备族。
- 支持至少 5 个高频告警代码。
- 每条告警能自动生成上下文数据清单。
- 80% 以上告警能成功生成时间线。
- 80% 以上告警能输出 Top 3 根因候选。
- 每个根因候选至少包含 1 条支持证据。
- 每次分析都有明确数据缺失说明。
- 工程师可以提交最终根因和处置反馈。
- 系统能统计响应时间、采纳率、命中率和复发率。

## 18. 后续可拆分模块

建议后续按以下模块拆任务：

- 告警标准模型与接入层。
- 数据连接器与上下文收集策略。
- 事件时间线与关联引擎。
- 历史案例库与知识检索。
- 根因候选生成器。
- 处置建议生成器。
- 告警工作台前端。
- 人工反馈与评估体系。
- 权限、审计与安全边界。
- 运营看板与效果评估。

## 19. 结论

本产品应先从设备告警响应切入，因为设备告警边界清楚、价值直接、数据链路相对明确。随后扩展到良率异常和工艺参数异常，最终沉淀为通用异常告警 Agent 平台。

产品成败的关键不在于 Agent 是否能“聊天”，而在于它能否稳定完成四件事：

1. 自动收集正确的数据。
2. 建立可信的事件关联。
3. 输出带证据的根因候选。
4. 形成工程师反馈闭环。

只要这四件事跑通，后续就可以逐步扩展设备类型、告警类型和分析深度。
